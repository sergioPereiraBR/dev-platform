Arquitetura Limpa

O DEV Platform adere à Arquitetura Limpa para separação de responsabilidades e independência de frameworks.

  • **Regra de Dependência:** Camadas internas não dependem de camadas externas. Fluxo claro: Domínio → Aplicação → Infraestrutura/Interfaces.
  • **Caminho Feliz:** Implementação do "caminho feliz" para use cases bem estruturados e isolados.
  • **Composição de Objetos:** O `composition_root.py` é o ponto central para instanciar e "conectar" as dependências, garantindo a Inversão de Controle.
  • **Interfaces e Adapters:** Uso de interfaces (`ports.py`, `interfaces.py`) para abstrair detalhes de infraestrutura, permitindo a fácil troca de implementações (ex: banco de dados, serviços externos).

Domain-Driven Design (DDD)

Foco no domínio de negócio, com elementos bem definidos para expressar a lógica complexa.

  • **Entidades:** (`entities.py`) Objetos com identidade e ciclo de vida, contendo estado e comportamento (ex: `User`).
  • **Value Objects:** (`value_objects.py`) Objetos imutáveis que descrevem características, sem identidade própria (ex: `Email`).
  • **Repositórios:** (`repositories.py`) Mecanismos para persistência e recuperação de agregados de domínio (ex: `UserRepository`).
  • **Serviços de Domínio:** (`services.py`) Lógica de negócio que não se encaixa naturalmente em Entidades ou Value Objects (ex: `UserService`).
  • **Exceções de Domínio:** (`exceptions.py`) Exceções customizadas para erros específicos do negócio, melhorando a clareza e testabilidade.

Princípios SOLID

Aderência aos cinco princípios para um design de software flexível e manutenível.

  • **SRP (Single Responsibility Principle):** Classes e módulos com uma única razão para mudar. (Ex: `UserCommands` focado em operações de usuário).
  • **OCP (Open/Closed Principle):** Aberto para extensão, fechado para modificação. (Ex: Uso de interfaces para permitir novas implementações sem alterar o código existente).
  • **LSP (Liskov Substitution Principle):** Subtipos devem ser substituíveis por seus tipos base. (Implícito no uso de interfaces).
  • **ISP (Interface Segregation Principle):** Interfaces pequenas e específicas para clientes. (Ex: `ports.py` com interfaces granulares).
  • **DIP (Dependency Inversion Principle):** Depender de abstrações, não de implementações. (Ex: Injeção de dependência via `composition_root.py`).

Boas Práticas de Programação

Além dos princípios, o projeto incorpora diversas práticas que elevam a qualidade do código.

  • **Estrutura de Diretórios:** Organização clara por camadas (`domain`, `application`, `infrastructure`, `interfaces`).
  • **Convenções de Nomenclatura:** Padrões consistentes para classes, funções e variáveis.
  • **Tipagem Estática:** Uso de `mypy` para aumentar a robustez e legibilidade do código.
  • **Programação Assíncrona:** Utilização de `async/await` para operações de I/O eficientes (`UserCommands`).
  • **Tratamento de Erros:** Exceções bem definidas e tratadas adequadamente em cada camada.
  • **DTOs (Data Transfer Objects):** Uso de DTOs para comunicação entre camadas, mantendo o domínio puro.
  • **Documentação:** Presença de arquivos como `README.md` e `mkdocs.yml` (indicando documentação gerada).
